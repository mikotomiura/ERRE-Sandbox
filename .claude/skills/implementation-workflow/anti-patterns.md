# Implementation Workflow — アンチパターン集

骨格を飛ばして実装に入ることで起きた実際の失敗パターン。

---

## アンチパターン 1: Step A をスキップして実装開始

**何が起きるか**:
- `architecture-rules` を確認せずに書いたため、`ui/` から `inference/` を直接 import してしまう
- 後で発覚してリファクタリングが必要になる

**正しい対処**:
必ず CLAUDE.md → docs/ → `.steering/requirement.md` の順で読んでから着手。

---

## アンチパターン 2: Step C の設計承認をスキップ

**何が起きるか**:
- 実装が 80% 終わったところで「実はアプローチが違う」とユーザーに言われて作業が無駄になる
- 設計の記録がないため、後で決定理由が分からなくなる

**正しい対処**:
`design.md` を書き、**ユーザーに明示的に承認を求めてから** 実装を始める。
「承認が得られるまで Step D に進まない」をルールとして徹底する。

---

## アンチパターン 3: tasklist を作らずに実装

**何が起きるか**:
- コンテキスト 50% を超えたときに「どこまで終わったか」が分からなくなる
- 複数の変更が混在してコミットの粒度が大きくなる

**正しい対処**:
Step D で必ず `tasklist.md` を作成する。
各タスクは 30 分以内で完了できる粒度に分割。
完了ごとにチェックを更新してコンテキスト圧縮に備える。

---

## アンチパターン 4: テストが通らないまま Step G に進む

**何が起きるか**:
- `code-reviewer` がテスト失敗の原因を「コードの問題」として報告し、混乱する
- レビュー結果が「テストを直す」話ばかりになってレビューの価値が下がる

**正しい対処**:
Step F で必ず `uv run pytest` を通してから Step G に進む。
テストが失敗した場合は `test-analyzer` で原因を特定してから修正する。

---

## アンチパターン 5: /fix-bug で回帰テストを先に書かない

**何が起きるか**:
- バグを直したつもりが同じバグが再発する
- 「なぜこのコードがある？」が後で分からなくなる

**正しい対処**:
`/fix-bug` では実装より先に、バグを再現する回帰テストを書く (TDD)。
テストが失敗することを確認してから実装する。

```
1. test_[bug_name].py を書く → pytest で失敗を確認
2. バグを修正
3. pytest で成功を確認
4. コミット (テストと修正を同一コミットに含める)
```

---

## アンチパターン 6: /refactor でテストが通る前提で進める

**何が起きるか**:
- リファクタリングと機能変更が混在してどちらのせいで壊れたか分からなくなる

**正しい対処**:
Step B の前に `uv run pytest` で全テスト グリーンを確認してから始める。
リファクタリング中は小さな単位で Step F (pytest) と交互に進める。

---

## アンチパターン 7: Refs: なしでコミット

**何が起きるか**:
- 3 ヶ月後に「なぜこの変更を入れたか」が分からなくなる
- `.steering/` の記録と git log の対応が取れなくなる

**正しい対処**:
すべてのコミットメッセージ末尾に `Refs: .steering/[YYYYMMDD]-[task-name]/` を付ける。
`git-workflow` Skill のテンプレートを使えば忘れない。

---

## アンチパターン 8: コンテキスト 50% 超で継続

**何が起きるか**:
- コンテキストが溢れて以前の設計決定を忘れる
- 誤ったコードを生成しても気づかない

**正しい対処**:
`/context` で使用率が 50% を超えたら次の区切り (テスト通過、コミット) で `/smart-compact` を実行。
タスク切り替え時は `/clear` で完全リセット。
