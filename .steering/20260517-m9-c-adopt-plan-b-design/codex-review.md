[HIGH-2] Corpus gate thresholds are overrideable at the production CLI
- Finding: I agree that de+en >= 0.60 plus de >= 0.30 can be treated as new Plan B achieved-corpus axes rather than a retroactive DI-5 trigger promotion. The problem is that the audit CLI exposes `--n-eff-min`, `--top-5-pct-max`, `--de-en-mass-min`, and `--de-mass-min`, and tests prove those overrides can flip a failing audit to pass. See `scripts/m9-c-adopt/audit_plan_b_corpus_stats.py:54-58`, `scripts/m9-c-adopt/audit_plan_b_corpus_stats.py:152-172`, `tests/test_audit_plan_b_corpus_stats.py:151-174`.
- Why it matters: A preregistered hard gate should not have an ordinary operator flag that can move thresholds post hoc. The emitted JSON records the changed thresholds, but the PR’s HIGH-3 discipline is to prevent threshold motion, not merely log it.
- Fix: Remove threshold override flags from `main()`. Keep constants as the only production thresholds; if test flexibility is needed, keep kwargs on the pure function but mark them test-only, or require an explicit non-ADOPT diagnostic mode that cannot emit `"plan_b_gate": "pass"`.
- Severity rationale: This is a direct path to post-hoc corpus-gate rescue.

[MEDIUM-2] “No addressee” is metadata-only, not output-filtered
- Finding: The prompt says not to address an interlocutor, but `filter_de_monolog` enforces only language, length, marker density, and trigram repetition. Accepted rows are then stored with `addressee_agent_id=None` / `addressee_persona_id=None` regardless of whether the text says “du”, “Sie”, “Frage”, or otherwise addresses a reader/questioner. See `scripts/m9-c-adopt/de_focused_monolog_collector.py:133-141`, `scripts/m9-c-adopt/de_focused_monolog_collector.py:347-418`, `scripts/m9-c-adopt/de_focused_monolog_collector.py:497-503`.
- Why it matters: Plan B’s corpus shape depends on true monolog examples. If addressed text is accepted as monolog, the weighting path will boost it as no-addressee data and can distort the Burrows/monolog signal.
- Fix: Add a fifth hard filter for addressee leakage, with German second-person/formal-address patterns and tests for accepted first-person monologs versus rejected addressed responses.
- Severity rationale: This is a research-quality risk in the collector, but it is locally fixable before the overnight run.

[LOW-1] Dry-run resume behavior contradicts its contract
- Finding: The CLI help says `--dry-run` skips resume, and the runbook says smoke dry-run does not preserve `pilot_state`; however `run_collection()` always reads `pilot_state` and always flushes it at completion. See `scripts/m9-c-adopt/de_focused_monolog_collector.py:540-542`, `scripts/m9-c-adopt/de_focused_monolog_collector.py:652-654`, `scripts/m9-c-adopt/de_focused_monolog_collector.py:720-727`, `.steering/20260517-m9-c-adopt-plan-b-design/g-gear-collection-runbook.md:51-54`.
- Why it matters: Reusing a smoke output path can contaminate acceptance-rate estimation with prior state.
- Fix: In dry-run mode, ignore existing state and skip `_flush_state`, or update the docs and tests to make dry-run resumable by design.
- Severity rationale: Operationally annoying, but unlikely to affect the final main run if paths are fresh.
