[HIGH-1] DA-14 Vendi CI gate is not preserved in the DA-15 standard rescore
- Finding: `rescore_vendi_alt_kernel.py` bootstraps Cohen's d for the standard window-level gate, but DA-14 stored and gated on the mean-difference CI (`diff_ci_95`) plus a separate Cohen's d point estimate. DA-15 then labels this d-CI as `natural_diff_hi` in the verdict path.
- Why it matters: HIGH-3 requires DA-14 thresholds to remain unchanged. In DA-14, `.steering/20260515-m9-c-adopt-retrain-v2-verdict/da14-verdict-v2-kant.json:8-15` records `diff_v2_minus_nolora`, `diff_ci_95`, and `cohens_d` separately, with `threshold_ci = "ci_upper < 0"`. DA-15's `_bootstrap_window_d_ci()` instead appends `_cohens_d(ra, rb)` and returns `lo` / `hi` over that d distribution (`scripts/m9-c-adopt/rescore_vendi_alt_kernel.py:142-163`), while `da15_verdict.py:73-90` treats `standard_bootstrap["hi"]` as the natural diff upper bound. This changes the statistical gate, even though the artifact says `diff_ci_upper_lt` (`.steering/20260516-m9-c-adopt-da15-impl/da15-rescore-e5-large-kant.json:44-48`).
- Fix: Split the standard bootstrap output into `diff_point`, `diff_lo`, `diff_hi`, and `cohens_d`. Compute `diff_*` from `mean(ra) - mean(rb)` and compute `cohens_d` once on the observed six windows. Update `da15_verdict.py` to gate on `cohens_d <= -0.5` and `diff_hi < 0`, regenerate all DA-15 rescore artifacts and verdict documents, and keep the old d-bootstrap out of the primary gate unless explicitly reported as exploratory.
- Severity rationale: HIGH because it directly violates the "DA-14 thresholds unchanged" requirement and could turn a future Plan A ADOPT/REJECT decision on the wrong CI statistic.

[HIGH-2] Pre-registration is documented but not enforced by the executable gate
- Finding: D-2 pins E5-large, BGE-M3, and MPNet revision SHAs (`.steering/20260516-m9-c-adopt-da15-impl/decisions.md:35-39`), and D-3 says exploratory encoders are excluded (`decisions.md:55-60`), but the implementation accepts any `--encoder`, does not load by pinned revision, and lets any encoder appearing in both `--rescore` and `--calibration` contribute to ADOPT eligibility.
- Why it matters: HIGH-1 was specifically about preventing post-hoc encoder shopping. `rescore_vendi_alt_kernel.py:251-260` and `da15_calibration_panel.py:301-310` only describe the primary candidates in help text; they do not restrict choices. `da15_verdict.py:267-288` builds `common = sorted(set(rescores) & set(calibrations))`, marks eligible encoders from that set, and declares ADOPT if quorum reaches 2, without checking a D-2 allowlist, revision SHA, library versions, or the pre-registration commit anchor. The BGE artifacts already show the audit gap: revision SHA is `<unavailable...>` and `library_versions` is `{}` (`da15-rescore-bge-m3-kant.json:1-5`, `da15-calibration-bge-m3.json:1-5`) while still entering the per-encoder verdict panel.
- Fix: Add a single D-2 allowlist constant or JSON manifest containing model id, revision SHA, allowed role (`primary` vs `regression`), and library versions. Pass the pinned `revision=` to `SentenceTransformer`, record the local/pre-registered commit SHA in outputs, and make calibration/rescore fail or mark `adopt_eligible=false` when revision/library metadata is unavailable or mismatched. In `da15_verdict.py`, ignore or hard-fail any encoder not in the primary allowlist; keep MPNet regression reported but never ADOPT-contributing.
- Severity rationale: HIGH because the current code path can admit an exploratory or wrong-snapshot encoder into the primary gate, defeating the pre-registration control that DA-15 was required to add.

[MEDIUM-1] Within-language calibration gate passes with an untested Japanese slice
- Finding: The calibration corpus includes Japanese examples, but the Japanese within-language AUC is `null` because it has only 3 Kant and 3 control items, and the pass predicate treats `auc is None` as passing.
- Why it matters: HIGH-2 requires the calibration panel to prove the encoder is not merely detecting language ID. `da15_calibration_panel.py:274-297` emits `auc: None` for slices below 5 examples per class, and `da15_calibration_panel.py:349-355` counts those null slices as `within_lang_pass`. The E5 artifact has `ja: { auc: null, n_kant: 3, n_control: 3 }` but still reports `within_language_pass: true` and `verdict: "PASS"` (`da15-calibration-e5-large.json:30-38`). That is a documented limitation, but it is not a successful within-language AUC gate for every language present in the corpus.
- Fix: Either build the calibration corpus so every included language has enough per-class mass for the CV gate, or exclude under-mass languages from the corpus before scoring and record them as out of scope. If a language remains in the corpus with insufficient mass, set `within_language_pass=false` or add an explicit pre-registered exception that prevents the encoder from being treated as fully HIGH-2-cleared.
- Severity rationale: MEDIUM because the current Plan A verdict is REJECT, so this does not rescue the PR today, but it overstates calibration coverage and would matter for any future ADOPT.

[MEDIUM-2] Language-balanced bootstrap implementation does not match its stricter documented method
- Finding: The rescore script documents language-balanced bootstrap as building language-pure windows and averaging per-language d (`rescore_vendi_alt_kernel.py:12-14`), but `_stratified_bootstrap_d()` instead draws a single mixed 100-utterance window with per-stratum quotas and computes one Vendi score over the mixed kernel (`rescore_vendi_alt_kernel.py:167-235`).
- Why it matters: A 50/50 mixed window is language-balanced in quota, but it still includes cross-language cosine pairs inside the Vendi kernel. That is a weaker control than the documented "compute Vendi per language, then average d" method and can still make the balanced score sensitive to cross-language embedding geometry.
- Fix: Either change the implementation to the documented method (per-language pure bootstrap windows, then average d/diff across languages with clear weighting), or change the documentation and verdict language to say this is quota-balanced mixed-window bootstrap. If the stricter method was the HIGH-2 acceptance criterion, regenerate artifacts with the stricter implementation.
- Severity rationale: MEDIUM because this affects the interpretation of the balancing audit, though the current artifacts fail either way.
